### 1. ¿Qué es InterVLAN Routing y por qué es importante?

Una **VLAN (Virtual Local Area Network)** es una red lógica que segmenta el tráfico dentro de una red física. Por diseño, los dispositivos en VLANs distintas **no pueden comunicarse entre sí** directamente, ya que cada VLAN es un dominio de broadcast independiente.

El **InterVLAN Routing** es el proceso mediante el cual se permite la comunicación entre dos o más VLANs diferentes, haciendo uso de un dispositivo de capa 3 (router o switch multicapa) que actúa como intermediario. Este dispositivo recibe el tráfico de una VLAN y lo enruta hacia la VLAN de destino.

#### ¿Por qué es importante?

En redes reales, una empresa típicamente tiene múltiples VLANs para separar departamentos (ej. VLAN 10 para Ventas, VLAN 20 para RRHH, VLAN 30 para Servidores). Sin InterVLAN Routing:

- Un equipo de Ventas **no podría acceder** a un servidor en la VLAN de Servidores.
- Los departamentos estarían completamente aislados, lo cual es inoperante en la práctica.
- Los recursos compartidos (impresoras, bases de datos, internet) serían inaccesibles entre VLANs.

Con InterVLAN Routing, se mantiene la **segmentación de seguridad** (cada VLAN sigue siendo su propio dominio de broadcast, reduciendo colisiones y tráfico innecesario) pero se habilita la comunicación controlada y enrutada entre ellas.

### 2. Las 3 formas de implementar InterVLAN Routing

#### 2.1 Heredado (Legacy InterVLAN Routing)

**¿Cómo funciona?**
En este método, cada VLAN tiene una **conexión física dedicada** entre el switch y el router. Es decir, si existen 3 VLANs, se necesitan 3 puertos físicos en el router y 3 cables hacia el switch.

**Funcionamiento:**
- Cada puerto del router se configura con una dirección IP que sirve como gateway de esa VLAN.
- El switch tiene los puertos en modo **Access** (un puerto por VLAN).
- El router enruta el tráfico recibido por la interfaz de VLAN-A hacia la interfaz correspondiente a VLAN-B.

**Ventajas:**
- Configuración simple.
- No requiere configuración de trunks.

**Desventajas:**
- **Muy ineficiente**: requiere un puerto físico del router por cada VLAN.
- No es escalable: un router con 4 puertos solo podría manejar 4 VLANs.
- Alto costo en hardware si hay muchas VLANs.
- Prácticamente en desuso en redes modernas.

**Escenario de uso:** Redes muy pequeñas o entornos de laboratorio educativo.

---

#### 2.2 Router-on-a-Stick (RoaS)

**¿Cómo funciona?**
En este método, se usa **un único enlace físico** entre el router y el switch, pero ese enlace se configura como **trunk** (tronco). Un enlace trunk puede transportar tráfico de múltiples VLANs usando el protocolo **802.1Q**, que etiqueta cada frame con el ID de su VLAN.

En el router, la interfaz física se divide en **subinterfaces virtuales** (una por VLAN). Cada subinterfaz se configura con:
- El encapsulamiento `dot1q <VLAN_ID>` para identificar qué VLAN le corresponde.
- Una dirección IP que actúa como gateway de esa VLAN.

**Ejemplo de configuración (referencia):**
```
Router(config)# interface GigabitEthernet0/0.10
Router(config-subif)# encapsulation dot1Q 10
Router(config-subif)# ip address 192.168.10.1 255.255.255.0

Router(config)# interface GigabitEthernet0/0.20
Router(config-subif)# encapsulation dot1Q 20
Router(config-subif)# ip address 192.168.20.1 255.255.255.0
```

**Ventajas:**
- Solo se necesita **1 cable y 1 puerto** físico en el router, sin importar cuántas VLANs haya.
- Mucho más económico y práctico que el método heredado.
- Fácil de agregar nuevas VLANs (solo se agrega una subinterfaz).

**Desventajas:**
- El enlace físico único puede convertirse en un **cuello de botella** si el tráfico entre VLANs es muy alto, ya que todo el tráfico enrutado pasa por ese único cable.
- El router tiene que procesar cada paquete entre VLANs (no es tan rápido como un switch capa 3).

**Escenario de uso:** Redes medianas donde el tráfico interVLAN no es excesivamente alto. Muy común en entornos académicos y empresas pequeñas/medianas.

---

#### 2.3 Switch Capa 3 (Layer 3 Switch / Multilayer Switch)

**¿Cómo funciona?**
Un **switch de capa 3** (también llamado switch multicapa) integra capacidades de enrutamiento directamente en el hardware del switch. En lugar de depender de un router externo, el propio switch enruta el tráfico entre VLANs usando **Interfaces Virtuales de Switch (SVI - Switched Virtual Interface)**.

Cada SVI es una interfaz lógica asociada a una VLAN con una dirección IP, que actúa como gateway para los hosts de esa VLAN. El enrutamiento se realiza en hardware (mediante chips ASIC), lo que lo hace extremadamente rápido.

**Ejemplo de configuración (referencia):**
```
Switch(config)# ip routing                     ← habilita el enrutamiento
Switch(config)# interface vlan 10
Switch(config-if)# ip address 192.168.10.1 255.255.255.0

Switch(config)# interface vlan 20
Switch(config-if)# ip address 192.168.20.1 255.255.255.0
```

**Ventajas:**
- **Rendimiento muy alto**: el enrutamiento ocurre en hardware, no en software.
- Elimina el cuello de botella del Router-on-a-Stick.
- Escalable para grandes redes con muchas VLANs.
- Reduce la necesidad de equipos adicionales (un solo dispositivo cumple la función de switch y router).

**Desventajas:**
- Mayor **costo inicial** comparado con un switch capa 2 + router.
- Configuración algo más compleja.

**Escenario de uso:** Redes empresariales, data centers, redes con alto volumen de tráfico interVLAN donde el rendimiento es crítico.

### 3. ¿Qué es un gateway predeterminado y por qué es un punto crítico?

#### Definición

El **gateway predeterminado** (default gateway) es la dirección IP del dispositivo de capa 3 (router o switch capa 3) al cual un host envía todos los paquetes destinados a redes que no conoce directamente. En otras palabras, es la "puerta de salida" de una red local hacia el exterior.

Cuando un host (por ejemplo, una PC con IP `192.168.10.50`) quiere comunicarse con un servidor en otra red (por ejemplo `192.168.20.100`), el host sabe que esa IP no está en su misma red, por lo que envía el paquete al gateway predeterminado (ej. `192.168.10.1`) para que este lo enrute al destino correcto.

#### ¿Por qué es un punto crítico?

El gateway predeterminado es un **único punto de fallo (Single Point of Failure - SPOF)**. Si el router que actúa como gateway falla:

- **Todos los hosts** de esa VLAN/subred pierden inmediatamente el acceso a cualquier otra red.
- Se pierde el acceso a Internet, a servidores en otras VLANs y a cualquier recurso externo.
- Los hosts no pueden redirigirse automáticamente a otro router, porque solo tienen configurado un gateway.
- La red queda completamente **aislada del exterior**.

#### Relación con la Redundancia

Para eliminar este punto crítico, se implementan **protocolos de redundancia de gateway** (HSRP, VRRP, GLBP). Estos protocolos permiten que dos o más routers compartan una **dirección IP virtual** que actúa como el gateway. Los hosts configuran esa IP virtual como su gateway. Si el router principal falla, otro router del grupo asume inmediatamente la responsabilidad, de forma transparente para los hosts.

Esto convierte el gateway de un SPOF en un componente de **alta disponibilidad**.

### 4. HSRP, VRRP y GLBP explicados con mis palabras

#### 4.1 HSRP — Hot Standby Router Protocol

HSRP es un protocolo propietario de **Cisco** que permite que dos o más routers trabajen en conjunto para proporcionar un gateway redundante a los hosts de una red.

El concepto central es: los routers del grupo eligen un **router activo** (Active) y un **router en espera** (Standby). Ambos comparten una **dirección IP virtual** y una **dirección MAC virtual**. Los hosts de la red configuran esa IP virtual como su gateway. Solo el router activo responde al tráfico real.

Si el router activo falla, el router en espera lo detecta (porque deja de recibir mensajes "hello" del activo) y **asume el rol activo** en pocos segundos, tomando la IP y MAC virtuales. Los hosts no se dan cuenta del cambio porque siguen usando la misma IP de gateway.

**Características clave:**
- Propietario de Cisco (solo funciona en equipos Cisco).
- El router activo maneja TODO el tráfico; el standby espera sin procesar tráfico real.
- Usa mensajes multicast para comunicación entre routers del grupo.
- Permite configurar prioridad para decidir quién es el activo.
- Tiene versiones: HSRPv1 (IPv4) y HSRPv2 (IPv4 e IPv6).

---

#### 4.2 VRRP — Virtual Router Redundancy Protocol

VRRP es funcionalmente muy similar a HSRP, pero es un **estándar abierto** definido por el IETF (RFC 5798), lo que significa que puede funcionar en equipos de diferentes fabricantes (Cisco, Juniper, HP, etc.).

En VRRP, el router principal se llama **Master** y los demás son **Backup**. También comparten una IP virtual. Una diferencia notable es que en VRRP, el router con la IP virtual real de la interfaz **puede ser también el Master**, lo que simplifica la configuración. Si el Master falla, el Backup con mayor prioridad asume el rol de Master.

**Características clave:**
- Estándar abierto (compatible con múltiples fabricantes).
- El router principal se llama "Master"; los demás, "Backup".
- Funcionalmente equivalente a HSRP, pero interoperable.
- Solo un router maneja el tráfico a la vez (sin balanceo de carga nativo).
- Usa multicast para comunicación interna del grupo.

---

#### 4.3 GLBP — Gateway Load Balancing Protocol

GLBP es un protocolo propietario de **Cisco** que va más allá de la simple redundancia: además de proporcionar un gateway de respaldo, permite **balancear la carga del tráfico** entre múltiples routers simultáneamente.

Funciona así: todos los routers del grupo comparten **una sola IP virtual** (el gateway que configuran los hosts), pero cada router tiene su propia **dirección MAC virtual distinta**. Un router especial llamado **AVG (Active Virtual Gateway)** asigna diferentes MACs virtuales a los hosts respondiendo las solicitudes ARP de forma rotativa (round-robin u otros algoritmos). Así, diferentes hosts envían su tráfico a diferentes routers, logrando un uso equilibrado de todos los routers disponibles.

Si algún router falla, el AVG simplemente deja de asignar esa MAC a nuevos hosts, y el tráfico se redistribuye entre los routers restantes.

**Características clave:**
- Propietario de Cisco.
- **Balanceo de carga real**: múltiples routers procesan tráfico simultáneamente.
- Un AVG coordina el grupo; cada router activo se llama **AVF (Active Virtual Forwarder)**.
- Soporta hasta 4 routers simultáneos en un grupo.
- Es el protocolo más eficiente de los tres en uso de recursos.

---

### 5. Diferencias entre los protocolos

#### 5.1 HSRP vs VRRP

| Característica | HSRP | VRRP |
|---|---|---|
| **Origen / Estándar** | Propietario de Cisco | Estándar abierto (IETF RFC 5798) |
| **Compatibilidad** | Solo equipos Cisco | Cualquier fabricante |
| **Nombre del router principal** | Active | Master |
| **Nombre del router secundario** | Standby | Backup |
| **IP virtual** | Siempre diferente a la IP real de la interfaz | Puede coincidir con la IP real de la interfaz |
| **Balanceo de carga** | No nativo (se puede simular con grupos) | No nativo (se puede simular con grupos) |
| **Versiones** | HSRPv1 y HSRPv2 | VRRPv2 (IPv4) y VRRPv3 (IPv4/IPv6) |
| **Comunicación interna** | Multicast 224.0.0.2 (v1) / 224.0.0.102 (v2) | Multicast 224.0.0.18 |

**En resumen:** HSRP y VRRP son funcionalmente casi idénticos. La diferencia principal es que HSRP es exclusivo de Cisco y VRRP es un estándar que funciona en redes multi-fabricante. Si tu red es 100% Cisco, cualquiera de los dos funciona. Si tienes equipos mezclados, **debes usar VRRP**.

---

#### 5.2 HSRP/VRRP vs GLBP

| Característica | HSRP / VRRP | GLBP |
|---|---|---|
| **Balanceo de carga** | ❌ No (solo uno activo a la vez) | ✅ Sí (múltiples routers activos simultáneos) |
| **Uso de routers** | Solo el activo procesa tráfico; los demás esperan | Todos los routers del grupo procesan tráfico |
| **Eficiencia de recursos** | Baja (routers de respaldo están "ociosos") | Alta (todos los routers trabajan) |
| **Complejidad** | Baja-media | Media-alta |
| **Número de IPs virtuales** | 1 IP virtual por grupo | 1 IP virtual, múltiples MACs virtuales |
| **Compatibilidad** | VRRP es multi-fabricante; HSRP es Cisco | Solo Cisco |
| **Ideal para** | Redes donde la redundancia es suficiente | Redes con alto tráfico donde se necesita balanceo |

**En resumen:** HSRP y VRRP proporcionan **redundancia** (si el principal falla, el respaldo toma el control). GLBP proporciona **redundancia + balanceo de carga** (todos los routers trabajan activamente, no hay ninguno ocioso). GLBP es más eficiente pero también más complejo de configurar.

### 6. ¿Qué ocurre si falla el router principal en una red sin redundancia?

Cuando el router principal (y único gateway) falla en una red sin redundancia, las consecuencias son inmediatas y severas:

**Impacto técnico:**
- Todos los hosts de la red **pierden conectividad** con cualquier red externa (incluyendo Internet y otras VLANs).
- Los hosts siguen enviando sus paquetes a la IP del gateway (que ya no responde), pero ningún dispositivo los procesa ni enruta.
- El protocolo ARP de los hosts sigue guardando en caché la MAC del router fallido durante un tiempo, intentando en vano comunicarse con él.
- La red local (dentro de la misma VLAN/subred) sigue funcionando, pero nada que requiera salir de esa subred funciona.

**Impacto operacional:**
- Se interrumpe el acceso a Internet para todos los usuarios.
- Los sistemas que dependen de comunicación entre VLANs (bases de datos, servidores de aplicaciones, impresoras en otra VLAN) quedan inaccesibles.
- En entornos empresariales, esto puede significar pérdida de productividad total, pérdida de ventas o incluso riesgos de seguridad.
- El tiempo de recuperación depende de cuánto tarde el equipo de TI en detectar el problema y reemplazar o reiniciar el router.

**Conclusión:** Sin redundancia, un solo fallo de hardware puede dejar a toda la organización sin conectividad hasta que el equipo sea reparado o reemplazado manualmente. Por eso los protocolos como HSRP, VRRP y GLBP son fundamentales en redes de producción.

---

### 7. ¿Qué protocolo elegirías en cada escenario?

#### Red pequeña → **HSRP (o VRRP)**

**Justificación:** En una red pequeña (oficina con 10-50 usuarios, pocos servicios), el tráfico interVLAN es moderado y el presupuesto es limitado. HSRP es suficiente para garantizar que si el router principal falla, el de respaldo tome el control rápidamente. La simplicidad de configuración de HSRP es ideal cuando no hay un equipo de red especializado. Si los equipos no son todos Cisco, se usaría VRRP. El balanceo de carga de GLBP sería un exceso para este escenario.

#### Red empresarial → **VRRP (o HSRP si es 100% Cisco)**

**Justificación:** En una red empresarial, es común tener equipos de múltiples fabricantes (Cisco, Juniper, etc.). VRRP, al ser un estándar abierto, garantiza compatibilidad sin importar el fabricante. Proporciona la redundancia necesaria para cumplir SLAs (Service Level Agreements) de alta disponibilidad. Se pueden crear múltiples grupos VRRP (uno por VLAN) para tener redundancia granular.

#### Red con alto tráfico → **GLBP**

**Justificación:** En redes con alto volumen de tráfico (data centers, campus universitarios, redes de ISPs), tener routers de respaldo "ociosos" esperando es un desperdicio de recursos. GLBP permite que todos los routers procesen tráfico simultáneamente, distribuyendo la carga y maximizando el throughput total. Si un router falla, los demás absorben su tráfico. Es la solución más eficiente cuando se necesita tanto alta disponibilidad como máximo rendimiento.

#### ORDEN 1 — Switch SW1

```

! Un puerto trunk transporta tráfico de múltiples VLANs etiquetado con 802.1Q
```

#### ORDEN 2 — Router 1 (R1) — HSRP Active

R1 se configura como el router principal (Active) de HSRP. Usaremos subinterfaces para hacer Router-on-a-Stick: una sola interfaz física dividida en subinterfaces, una por VLAN.

```
! ── Entrar al modo privilegiado ──
Router> enable
Router# configure terminal

! ── Cambiar nombre ──
Router(config)# hostname R1

! ── PASO 1: Activar la interfaz física principal ──
! La interfaz física NO tiene IP, solo se enciende. Las IPs van en las subinterfaces.
R1(config)# interface GigabitEthernet0/0
R1(config-if)# description Enlace-Trunk-hacia-SW1
R1(config-if)# no shutdown
R1(config-if)# exit

! ── PASO 2: Crear subinterfaz para VLAN 10 ──
! El número después del punto (0/0.10) es solo una convención; usamos el ID de la VLAN
R1(config)# interface GigabitEthernet0/0.10
R1(config-subif)# description Gateway-VLAN10-Ventas
R1(config-subif)# encapsulation dot1Q 10
! "encapsulation dot1Q 10" le dice a esta subinterfaz que procese los frames etiquetados con VLAN 10
R1(config-subif)# ip address 192.168.10.2 255.255.255.0
! Esta es la IP REAL de R1 en VLAN 10 (no la virtual de HSRP)
R1(config-subif)# exit

! ── PASO 3: Crear subinterfaz para VLAN 20 ──
R1(config)# interface GigabitEthernet0/0.20
R1(config-subif)# description Gateway-VLAN20-Administracion
R1(config-subif)# encapsulation dot1Q 20
R1(config-subif)# ip address 192.168.20.2 255.255.255.0
! Esta es la IP REAL de R1 en VLAN 20
R1(config-subif)# exit

! ── PASO 4: Configurar HSRP en la subinterfaz de VLAN 10 ──
R1(config)# interface GigabitEthernet0/0.10
R1(config-subif)# standby 10 ip 192.168.10.1
! "standby 10" = grupo HSRP número 10 (por convención usamos el mismo ID de VLAN)
! "ip 192.168.10.1" = la IP VIRTUAL compartida (la que configurarán los hosts como gateway)
R1(config-subif)# standby 10 priority 110
! Prioridad 110 > 100 (R2), por eso R1 gana la elección y se vuelve Active
R1(config-subif)# standby 10 preempt
! "preempt" = si R1 cae y vuelve a subir, recupera automáticamente el rol de Active
R1(config-subif)# standby 10 version 2
! HSRPv2 soporta IPv6 y más grupos; es la versión recomendada
R1(config-subif)# exit

! ── PASO 5: Configurar HSRP en la subinterfaz de VLAN 20 ──
R1(config)# interface GigabitEthernet0/0.20
R1(config-subif)# standby 20 ip 192.168.20.1
! IP virtual para VLAN 20
R1(config-subif)# standby 20 priority 110
R1(config-subif)# standby 20 preempt
R1(config-subif)# standby 20 version 2
R1(config-subif)# exit

! ── PASO 6: (Opcional) Guardar configuración ──
R1(config)# end
R1# write memory
```

**¿Qué hicimos en R1?**
- Activamos la interfaz física sin IP (la IP va en las subinterfaces).
- Creamos `Gi0/0.10` para manejar tráfico de VLAN 10 con su IP real `192.168.10.2`.
- Creamos `Gi0/0.20` para manejar tráfico de VLAN 20 con su IP real `192.168.20.2`.
- Configuramos HSRP en cada subinterfaz con una IP virtual (`192.168.10.1` y `192.168.20.1`) que los hosts usarán como gateway.
- Prioridad 110 asegura que R1 sea el **Active** (el que realmente atiende el tráfico).
- `preempt` permite que R1 recupere el rol automáticamente si se reinicia.

#### ORDEN 3 — Router 2 (R2) — HSRP Standby

R2 es casi idéntico a R1, excepto que tiene **menor prioridad** (100 en lugar de 110) y **no tiene preempt** (o si lo tiene, no importa porque pierde la elección de todas formas). R2 solo toma control si R1 falla.

```
! ── Entrar al modo privilegiado ──
Router> enable
Router# configure terminal

! ── Cambiar nombre ──
Router(config)# hostname R2

! ── PASO 1: Activar la interfaz física ──
R2(config)# interface GigabitEthernet0/0
R2(config-if)# description Enlace-Trunk-hacia-SW1
R2(config-if)# no shutdown
R2(config-if)# exit

! ── PASO 2: Subinterfaz para VLAN 10 ──
R2(config)# interface GigabitEthernet0/0.10
R2(config-subif)# description Gateway-Backup-VLAN10
R2(config-subif)# encapsulation dot1Q 10
R2(config-subif)# ip address 192.168.10.3 255.255.255.0
! IP REAL de R2 en VLAN 10 (diferente a la de R1 y a la IP virtual)
R2(config-subif)# exit

! ── PASO 3: Subinterfaz para VLAN 20 ──
R2(config)# interface GigabitEthernet0/0.20
R2(config-subif)# description Gateway-Backup-VLAN20
R2(config-subif)# encapsulation dot1Q 20
R2(config-subif)# ip address 192.168.20.3 255.255.255.0
! IP REAL de R2 en VLAN 20
R2(config-subif)# exit

! ── PASO 4: Configurar HSRP en VLAN 10 ──
R2(config)# interface GigabitEthernet0/0.10
R2(config-subif)# standby 10 ip 192.168.10.1
! MISMA IP virtual que R1 (así funcionan en el mismo grupo HSRP)
R2(config-subif)# standby 10 priority 100
! Prioridad 100 < 110 (R1), por eso R2 pierde la elección y queda como Standby
R2(config-subif)# standby 10 version 2
! Debe coincidir con la versión configurada en R1
R2(config-subif)# exit

! ── PASO 5: Configurar HSRP en VLAN 20 ──
R2(config)# interface GigabitEthernet0/0.20
R2(config-subif)# standby 20 ip 192.168.20.1
R2(config-subif)# standby 20 priority 100
R2(config-subif)# standby 20 version 2
R2(config-subif)# exit

! ── PASO 6: Guardar ──
R2(config)# end
R2# write memory
```

**¿Qué hicimos en R2?**
- Misma estructura que R1, pero con IPs reales diferentes (`.3` en lugar de `.2`).
- La IP virtual HSRP es **la misma** que en R1 (`192.168.10.1` y `192.168.20.1`), porque ambos pertenecen al mismo grupo HSRP.
- Prioridad 100 hace que R2 sea automáticamente el **Standby** (espera sin atender tráfico).
- R2 monitorea a R1 cada 3 segundos y toma control solo si R1 deja de responder.

#### ORDEN 4 — PC-A y PC-B (Configuración de las PCs)

Las PCs se configuran al final porque necesitan apuntar a la IP virtual de HSRP como gateway, y esa IP ya debe estar funcionando en los routers.

**PC-A (pertenece a VLAN 10):**

En Packet Tracer, hacer clic en PC-A → pestaña **Desktop** → **IP Configuration**:

```
IP Address    : 192.168.10.10
Subnet Mask   : 255.255.255.0
Default Gateway: 192.168.10.1     ← IP VIRTUAL de HSRP (no la real de R1 ni R2)
DNS Server    : (dejar vacío o poner 8.8.8.8)
```

**PC-B (pertenece a VLAN 20):**

```
IP Address    : 192.168.20.10
Subnet Mask   : 255.255.255.0
Default Gateway: 192.168.20.1     ← IP VIRTUAL de HSRP para VLAN 20
DNS Server    : (dejar vacío o poner 8.8.8.8)
```

> **Punto clave:** Los hosts nunca deben usar como gateway las IPs reales de R1 (`.2`) ni de R2 (`.3`). Siempre deben apuntar a la **IP virtual** (`.1`). Si apuntaran a la IP real de R1 y R1 falla, la redundancia no funcionaría porque R2 no tiene esa IP.

#### Resumen del orden de configuración

| Paso | Dispositivo | Qué se configura | Por qué en ese orden |
|---|---|---|---|
| 1 | **SW1** | VLANs, puertos trunk, puertos access | El switch debe tener las VLANs antes que los routers puedan usarlas |
| 2 | **R1** | Subinterfaces, IPs reales, HSRP Active | R1 se configura primero para que tome el rol Active al inicio |
| 3 | **R2** | Subinterfaces, IPs reales, HSRP Standby | R2 se une al grupo HSRP ya creado por R1 |
| 4 | **PC-A / PC-B** | IP, máscara, gateway (IP virtual HSRP) | Los hosts se configuran al final, cuando el gateway ya está activo |
| 5 | **Verificación** | `show standby`, `show vlan`, `ping` | Confirmar que todo funciona como se espera |

#### Explicación del funcionamiento del diseño con HSRP

1. **En condiciones normales:** Los hosts de VLAN 10 tienen configurado `192.168.10.1` como gateway. Este corresponde a la IP virtual de HSRP. El Router 1 (Active, prioridad 110) responde a las solicitudes ARP para esa IP y procesa todo el tráfico.

2. **Fallo del Router 1:** El Router 2 deja de recibir los mensajes "Hello" de R1 (enviados cada 3 segundos por defecto). Tras el tiempo de espera (Hold Time, por defecto 10 segundos), R2 asume el rol de **Active** y comienza a responder a la IP virtual `192.168.10.1`.

3. **Recuperación de Router 1:** Si R1 se recupera y está configurado con `preempt` (preferencia para recuperar el rol), R1 vuelve a ser Active automáticamente por tener mayor prioridad (110 > 100).

4. **Transparencia para los hosts:** Durante todo el proceso, los hosts nunca cambian su configuración de gateway. Siguen apuntando a `192.168.10.1`. El cambio de router es completamente transparente para ellos.
